# Settings Refactor Plan (Stability-First)

Last updated: 2026-02-12  
Status: `IN_PROGRESS`

## Goal
Refactor Settings so navigation and section rendering cannot regress when individual pages are edited.

## Baseline Gate (Blocking)
1. No net-new UI features are allowed until settings baseline is complete.
2. Settings baseline is a required sub-track of the global UI baseline (see `ui_rewrite_plan.md` and `DESIGN_SYSTEM_RULES.md`).
3. Non-settings UI baseline must also complete before net-new UI feature work starts.
4. Settings-baseline complete means:
   - all settings pages are migrated to shared token + primitive contracts,
   - all settings controls (tabs, selects, checkboxes, inputs, buttons, list rows, section panels) match shared behavior,
   - smoke checklist passes in both `pnpm dev` and `pnpm run runserver` (after `pnpm build`).

## Mandatory Guardrails
1. Behavior freeze is defined in `SETTINGS_BEHAVIOR_CONTRACT.md`.
2. Post-slice verification is defined in `SETTINGS_SMOKE_CHECKLIST.md`.
3. During `Phase 1` and `Phase 2`, no business logic changes are allowed.

## Why This Track Exists
Recent regressions came from mixing three concerns in one slice:
1. token/visual updates,
2. primitive changes,
3. page logic/state changes.

This plan separates concerns and adds checkpoints after each phase.

## Scope
In scope:
1. Settings shell structure and page tab behavior.
2. Shared settings primitives and token usage.
3. Migration away from legacy/dead settings flags and branches.
4. Desktop and mobile settings rendering parity.
5. Full settings-page and settings-control standardization before any further UI feature work.

Out of scope:
1. Non-settings chat layout rewrite.
2. Provider/backend behavior changes.
3. New feature design in settings pages.

## Execution Order (Mandatory)
1. Tokens.
2. Primitives.
3. Settings shell contract.
4. Page-by-page migration.
5. Automated checks and hardening.

No page-level redesign before steps 1-3 are locked.

## Slice Protocol (Mandatory)
For every settings slice:
1. Change one scoped concern only.
2. Run `pnpm check` (and `pnpm check:settings-smoke` for settings pages/shell).
3. Use dev-first loop (`pnpm dev`) for fast visual validation.
4. Run `pnpm build` + `pnpm run runserver` at checkpoint slices, when server-mode behavior is being validated, or before final server-mode signoff.
5. Run all checks from `SETTINGS_SMOKE_CHECKLIST.md` at the appropriate cadence (per-slice dev checks; batched server-mode checks).
6. If any regression is found, revert the slice before proceeding.
7. Update corresponding settings rows in `UI_BASELINE_MATRIX.md`.
